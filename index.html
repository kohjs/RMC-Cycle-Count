🔧 CK3 Handheld Device - Reject Cycle Count System (5-Phase Complete Workflow)

🚨 Error Scenarios Handled

  • Wrong Pallet Scanned: Pallet exists but not in current count list → Display error, prompt rescan
  • Invalid Bundle Scanned: Bundle doesn't belong to current pallet's sample list → Rejected, no action taken
  • Duplicate Bundle Scan: Same bundle already scanned on this pallet → Warning displayed, continue scanning
  • System Block: Sample bundles remain in list → Cannot proceed to next pallet until all accounted for
  • MTF Not Found: Refresh pressed but no MTF request exists → Prompt operator to rescan or file discrepancy
  • Missing Bundles: Bundles not found after rescan → Press Discrepancy button, scan badge, enter reason
  • Incomplete Cycle: Not all customers' pallets counted → Generate next cycle preview, loop back to Phase 1

Phase 1: System Initialization & Scheduling (Backend System)

⚙️ This phase runs on the backend system, not on the CK3 handheld device

🖥️ Backend System Processes

The following processes are automated and occur on the backend system before the operator uses the CK3 device:

📅 Automated Scheduling

  • Generate 2-3 customers per month
  • Group pallets by customer
  • Sort by age (older pallet numbers first)
  • Select 5% sample of aging pallets
  • Create cycle count ID and list

📦 Bundle Sampling

  • Select sample bundles from sampled pallets
  • Record Pallet-Lot-Bundle-Location
  • Store customer info & reject types
  • Generate master sample list

💾 Database Recording

  • Record full cycle count start date
  • Link to existing RMC data
  • Store complete cycle count data
  • Set status to "Pending"

📧 Email Notifications

  • Send initial alert to RMC team
  • Include count ID, customer, due date
  • List pallet count and bundle count
  • Send weekly reminders until completed

⏳ Status: Pending (Awaiting Manual Trigger)

Once Phase 1 is complete, the operator can begin Phase 2 on the CK3 handheld device

📊 Complete Screen Flow Summary

Overview of all CK3 screens in the pallet-based bundle validation process

Screen Flow Breakdown

Phase 1: System Initialization & Scheduling (Backend Only - No CK3 Screens)

  • ⚙️ Automated scheduling (2-3 customers/month, 5% aging pallets)
  • ⚙️ Email notification sent to RMC team
  • ⚙️ Database update with cycle count list
  • ⚙️ All processes run on backend system, not CK3 device

Phase 2: Primary Count Execution (6 screens)

  • ✓ Screen 1: Scan badge number
  • ✓ Screen 2: Enter cycle count ID
  • ✓ Screen 3: View bundle list (grouped by pallet)
  • ✓ Screen 4: Navigate and scan pallet ID
  • ✓ Screen 5: Error - Wrong pallet for this count
  • ✓ Screen 6: Success - Pallet validated, ready for bundle scanning

Phase 3: Bundle Scanning (3 screens)

  • ✓ Screen 7: Scan bundles from sample list (updates in-place with notification bar when bundle scanned)
  • ✓ Screen 8: Duplicate bundle warning (dismiss returns to Screen 7)
  • ✓ Screen 9: Example of Screen 7 after successful scan (notification bar + updated list)
  • ✓ Screen 10: Pallet complete - All bundles checked

Note: Screen 9 is not a separate screen - it shows how Screen 7 updates after a bundle is scanned. The workflow stays on Screen 7 with quiet in-place updates.

Phase 4: Pallet Completion Verification (6 screens)

  • ✓ Screen 11: System block - Bundles remaining in list
  • ✓ Screen 12: Refresh button - Checking MTF requests
  • ✓ Screen 13: MTF found - Bundle replacement
  • ✓ Screen 14: No MTF - Prompt rescan or discrepancy
  • ✓ Screen 15a: Badge authorization for discrepancy
  • ✓ Screen 15b: Select reason and enter comments

Phase 5: Completion & Coverage Check (3 screens)

  • ✓ Screen 16: Discrepancy report (if any) with email notification
  • ✓ Screen 17: Completion rates (current cycle + total cycles)
  • ✓ Screen 18: Next cycle preview and coverage status

Total: 18 CK3 Screens (Phases 2-5) + Phase 1 Backend

Complete System Workflow: Backend scheduling → Sample-based presence validation on CK3 → MTF handling → Discrepancy management → Completion tracking
Key Features: 5% aging pallet sampling, system blocking, refresh mechanism, badge authorization, MTF replacement, completion rates, next cycle preview
CK3 Workflow: Phase 1 runs on backend system; Phases 2-5 run on CK3 handheld device with in-place updates and notification bars
Looping Structure: System continues until 100% pallet coverage across all customers achieved
Final Status: Cycle Complete with detailed reports and coverage tracking
